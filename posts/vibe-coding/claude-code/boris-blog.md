刚刷到 Claude Code 的作者 Boris 发的一条长推，看得我背脊发凉。

说实话，自从 Claude Code 发布以来，我和大家一样，觉得自己已经玩得挺溜了——无非就是把终端交给 AI，让它帮我改改 Bug，写写脚本。我觉得这已经是生产力的飞跃了。

但看完 Boris 晒出的所谓“朴实无华”的个人配置，我才发现：**贫穷限制了我的想象力，我依然是在用“驴拉磨”的思维在驾驭一辆法拉利。**

他这哪里是在用一个编程工具？他简直是在指挥一个完整的工程战队。

这篇推文里藏着几个极具颠覆性的细节，我必须要把它们挖出来，跟大家好好聊聊。

#### 1. 既然是 AI，为什么要排队？
Boris 说他终端里常年开着 **5 个 Claude 进程**，浏览器里再挂 5-10 个。

看到这点我才猛拍大腿。是啊，我们这帮写代码的，习惯了单线程工作：写完 A 模块，再写 B 模块。但 AI 不用休息啊！

Boris 的玩法是：在这个窗口让 Claude 1 号去修 Bug，切个 Tab 让 Claude 2 号去重构旧代码，再让 3 号去写文档。利用系统通知来管理回调。**这就是把单兵作战瞬间变成了“多线程并发”。**

以前我们觉得等待模型生成代码是垃圾时间，现在看来，那是我们不懂得调度。

#### 2. “记忆”才是团队的核心资产
这可能是整篇分享里最值钱的一点。Boris 的团队维护了一个 `CLAUDE.md` 文件，**并把它存进了 Git 仓库**。

这就好比给 AI 发了一本“员工手册”。
每次 Claude 犯错，或者团队有了新的代码规范，他们不是口头纠正一下就完了，而是直接更新进 `CLAUDE.md`。

这意味着什么？这意味着团队里所有的“踩坑经验”都被数字化、持久化了。Claude 不再是一个每次重启都会失忆的实习生，而是一个越老越妖、越用越顺手的资深工程师。**这就是传说中的“复利效应”，代码在迭代，AI 的智商也在跟着项目一起迭代。**

#### 3. 宁愿慢，也要“聪明”
在模型选择上，Boris 直接摊牌了：他全盘使用 **Opus 4.5**（也就是那个带 thinking 过程的最强模型）。

哪怕 Opus 贵一点、慢一点，他也坚决不用更快的 Sonnet。为什么？因为对于复杂逻辑，Opus 一次就能做对。

这给了我很大的触动。我们往往太在意 AI 生成代码的那几秒钟延迟，却忽略了其实最浪费时间的是 AI 听不懂人话、反复修改来回拉扯的过程。**“一次做对”，才是最顶级的快。**

#### 4. 别当复读机，做个工具人管理者
Boris 的工作流里充满了“自动化”的哲学。

他发现自己如果一天内要重复输入某段 Prompt 几次，他就会立刻把它封装成一个 Slash Command（斜杠命令）。比如 `/commit-push-pr`，一条命令自动搞定 Git 状态检查、提交、推送到 PR。

更有意思的是，他甚至用 AI 来监管 AI。他写了 `code-simplifier`（代码简化器）和 `verify-app`（应用验证器）这种子 Agent。代码写完了？先别交，让子 Agent 跑一遍去格式化、去验证。

这才是把 AI 当成了真正的生产力接口，而不是一个聊天框。

#### 5. 信任是好的，但验证更重要
最后，也是最扎心的一点。Boris 强调：**一定要给 Claude 一个反馈闭环。**

他不会写完代码就直接跑，而是会给 Claude 提供一套测试工具（甚至包括 Chrome 插件来模拟浏览器点击）。Claude 写完代码，自己去跑测试，红灯了自己修，直到绿灯为止。

Boris 说，只要加上这个反馈环节，代码质量直接翻 2-3 倍。

细思极恐。这意味着未来的开发者，核心竞争力不再是“写代码”，而是“写测试”和“定义验证标准”。**只要你能定义清楚什么是“对的”，AI 就能帮你搞定剩下的所有过程。**

### 写在最后

看完 Boris 的分享，我把自己的终端配置全部删了重来。

我们正在经历一个从“Copilot（副驾驶）”到“Auto-Pilot（自动驾驶）”的转折点。很多人还在纠结 AI 会不会写错代码，而顶级的玩家已经开始思考如何构建一套系统，让 AI 在里面自我纠错、自我进化。

就像 Boris 说的，Claude Code 只是个底座。怎么去魔改它、怎么把它嵌入你的工作流，才是拉开差距的关键。

建议大家都去看看那个 `CLAUDE.md` 的思路，哪怕只是这一个改动，可能都会让你的开发体验天翻地覆。
