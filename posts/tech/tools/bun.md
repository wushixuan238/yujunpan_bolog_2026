咱们来聊聊怎么用 Bun 管理依赖。这东西最近火得一塌糊涂，速度快得让人怀疑人生。如果你想把手头的 npm 项目切过来，其实非常简单。

先把 Bun 装好，Windows 用户直接在 PowerShell 里跑那行 irm 命令，Mac 和 Linux 用户用 curl 就行。装好之后别急着弄，先把旧的 package lock json 文件和 node modules 目录全删了，免得残留文件打架。清理干净后敲一个 bun install，它会自动扫描你的 package json，生成它专用的锁文件。至于原本写好的 scripts 脚本，Bun 完美兼容，你甚至都不用改文件，直接把原来的 npm run 换成 bun run 就能跑。

要是开新坑就更省事了。想用 vite 或者 next app 这种框架，直接 bun create 加上模版名就能秒建项目。如果是自己从零开始写个小工具，建好文件夹进去跑个 bun init 也就完事了。平时装依赖习惯也不用怎么变，把 install 换成 add 就行，比如 bun add react。开发依赖记得加个大写的 D 参数，跟 npm 的逻辑是一模一样的。

很多朋友担心命令记不住，其实只要记住把 npx 换成 bunx，把 uninstall 换成 remove，其他的操作直觉上都是通用的。

不过有几个细节得由衷建议一下。Bun 默认生成的锁文件是二进制格式的，git diff 看不到内容，但这换来的是极致的解析速度。千万别在一个项目里混用 npm 和 bun 的命令，锁文件一旦打架，修起来很头大。还有就是它的安装速度真的快，因为它用了全局缓存，同样的包不用重复下载。

万一碰到安装报错，多半是缓存乱了，清空一下 bun 目录下的 cache 再试通常能解决。国内网络环境下，要是觉得拉包慢，记得设一下淘宝镜像源。做自动化部署的话，GitHub Actions 也有官方的 setup bun 插件，配置起来也就是几行代码的事。

真的推荐大家试一试，那种几秒钟装完上千个依赖的体验，试过之后就很难再忍受 npm 的进度条了。想看更细致的参数配置，直接去翻翻官网文档就好。
